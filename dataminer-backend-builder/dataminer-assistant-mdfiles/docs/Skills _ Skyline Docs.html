Skills

The Skills capability introduces the Agent Skills framework, allowing users to extend the Assistant's capabilities by adding custom skills as SKILL.md files. Skills are automatically discovered, validated, cached, and synchronized across cluster members.

Skills are available as of DataMiner Assistant 2.2.0. As of DataMiner Assistant 3.3.5 and DataMiner Web 10.6.8, skills can be created, updated, and deleted and assigned to specialized agents.

Overview

What are agent skills

Agent Skills are modular capabilities that package instructions, metadata, and optional resources (for example scripts, templates, and reference files). They let the assistant reuse domain-specific workflows and best practices across conversations instead of relying only on one-off prompt instructions.

Why use skills

  • Specialize the assistant for domain-specific tasks and internal processes.
  • Reduce repetition by authoring guidance once and reusing it automatically.
  • Compose multiple skills to support more complex end-to-end workflows.

How skills work

Skills use a filesystem-based model. Each skill is a folder containing SKILL.md and optional resource files. The assistant then loads information progressively:

  1. Metadata is always available for discovery.
  2. Instructions are loaded when a skill is triggered.
  3. Extra text-based resources are only read and passed along to the LLM when needed.

This progressive loading model keeps context usage efficient, because only relevant content is brought into the active context window.

Managing skills in the Assistant app

Skills can be created and managed through the Assistant app (available from DataMiner Web 10.6.8).

Anyone can view existing skills. Creating or editing skills requires the Modules > System configuration > Tools > Admin tools user permission.

Skills overview page

The Skills tab in the left sidebar of the Assistant app displays all available skills in a searchable list. Each skill is listed by name, allowing you to quickly find and select a skill to view or edit.

UI taken from DataMiner Web 10.6.8.

At the top of the page, a brief description explains what skills are: structured instructions that define how an agent can solve a specific task. They describe what to do and may involve tools, agents, or other skills.

From this page, you can:

  • Search for existing skills by name using the search bar.
  • Create a new skill by clicking the + New skill button. This button is only visible if you have the required Admin tools permission.
  • Open a skill by clicking its name in the list to view or edit its details.

Skill details page

When you open a skill, the details page displays all of its configurable fields:

UI taken from DataMiner Web 10.6.8.

Field Description
Name The unique identifier for the skill. Must be lowercase, may only contain letters, digits, and hyphens, and must not start or end with a hyphen. Maximum 64 characters.
Description A short summary of what the skill does and when it should be used. Maximum 1,024 characters. This description is used by the agent to decide when to invoke the skill.
Context The full skill instructions in Markdown format. This is the content that gets loaded into the agent's context window when the skill is triggered. It typically contains workflow steps, rules, constraints, and references to companion resources.

At the top of the details page, the following actions are available:

  • Save changes: Persists your edits.
  • Delete: Removes the skill entirely.
  • Cancel: Discards unsaved changes and returns to the overview.
Tip

The Description field is critical for skill discovery. Write it as a clear trigger condition so the agent knows exactly when to activate the skill. For example: "Use whenever questions arise around roadmap items, products, releases, teams, backlog, upcoming work, or completed work."

Note

Skills created or edited through the Assistant app are stored in the same location as skills added via the filesystem and are synchronized across all cluster members.

File storage

Whether created through the Assistant app or added manually, skills are stored on disk and automatically synchronized across all cluster members.

Location

Skills are located in the following directory and are automatically synchronized across the cluster:

  • DataMiner Assistant 2.3.7 and later
  • DataMiner Assistant 2.2.0 through 2.3.6

%ProgramData%\Skyline Communications\DataMiner Assistant\Synced Documents\Context\Custom\Skills

Skill Structure

A skill is always defined in its own folder, and called SKILL.md. Inside that folder, you can have many more text-based resources that can be used. They can be inside the same folder or inside a nested subfolder.

Skills/
|-- my-custom-skill/
|   |-- SKILL.md
|   |-- specialized-skill.md
|   `-- templates/
|       |-- template01.txt
|       |-- template02.txt
|   `-- playbooks
|       |-- playbook01.md
|       |-- playbook02.md

SKILL.md Format

  • YAML front matter with required fields: name and description.

  • Markdown body containing the skill instructions.

  • Validation rules:

    • name: Required. Maximum 64 characters. Must be lowercase and may only contain letters, digits, and hyphens. Cannot start or end with a hyphen, and cannot contain consecutive hyphens (--).
    • description: Required. Maximum 1,024 characters.
    • Folder name must match the skill name.

    These rules are enforced whether you create or update skills through the API or by adding .md files directly. If a field exceeds its limit or violates a naming rule, a descriptive error is returned indicating what is invalid.

Example SKILL.md

---
name: change-request-review
description: Reviews a proposed change request for completeness, risk, and rollout readiness using internal checklists and templates.
---

# Change Request Review

Use this skill when a user asks to review, validate, or improve a software change request before implementation.

## When To Use

- A change request needs a structured quality review.
- A user asks for risk analysis, missing details, or rollout validation.
- The request must follow internal release and testing standards.

## Inputs To Collect

- Change summary and business goal.
- Affected components, services, or repositories.
- Proposed rollout plan and rollback plan.
- Known constraints, deadlines, and dependencies.

## Workflow

1. Confirm scope and restate the requested change in one paragraph.
2. Check for missing required fields using `templates/change-request-required-fields.md`.
3. Evaluate risk using `playbooks/risk-evaluation.md` and assign `low`, `medium`, or `high` risk.
4. Validate rollout and rollback quality using `templates/rollout-checklist.md`.
5. Produce a final review with:
  - `Findings` (ordered by severity)
  - `Open Questions`
  - `Recommended Next Actions`

## Output Requirements

- Be specific and actionable.
- Reference concrete gaps instead of generic advice.
- If information is missing, ask targeted follow-up questions.
- Do not approve a request if rollback details are absent.

## Companion Resources

- `templates/change-request-required-fields.md`
- `templates/rollout-checklist.md`
- `playbooks/risk-evaluation.md`
