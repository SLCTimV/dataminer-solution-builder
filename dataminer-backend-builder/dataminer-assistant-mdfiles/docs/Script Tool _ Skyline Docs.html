Script Tool

As of DataMiner Assistant 2.3.16, Automation scripts can be discovered and executed by being prompted through the Assistant's Insight agent.

Scripts need to adhere to the following requirements before being eligible for execution:

  • A corresponding context file is created (see Context Files for Automation Scripts).
  • The feature flag for automation scripts is enabled.
Note

Prior to DataMiner Assistant 3.3.5, scripts needed to be located in the Automation\bot folder. As of version 3.3.5, script tools can run any script on the system.

Context Files for Automation Scripts

To describe available Automation scripts and guide LLM usage, add context files in:

C:\ProgramData\Skyline Communications\DataMiner Assistant\Synced Documents\Context\Custom\Scripts

Requirements:

  • Use one .md file per Automation script.
  • Include YAML front matter.
  • The name in front matter must match the Automation script name. Must be lowercase and may only contain letters, digits, and hyphens. Cannot start or end with a hyphen, and cannot contain consecutive hyphens (--).
  • sync is optional and defaults to true when omitted.

Execution Behavior

The sync front matter property controls how the AutomationScriptTool executes scripts:

  • sync: true: The LLM waits for the Automation script result.
  • sync: false: Trigger-and-forget execution. The LLM does not wait for the result.

CoreGateway applies a timeout of 50 seconds for Automation script execution.

Example

---
name: maintenance
description: Script to perform Maintenance on a certain device
sync: true
inputArguments:
  - name: "elementName"
    example: "MyComputer"
---

## Usage Example

**User**: "I want to start maintenance on MyComputer"

## Error Handling

**Missing elementName**: Ask user to provide valid value

Technical details

Connection behavior

Impersonated connections are cached and remain open for at least 15 minutes after they were created. When the connection is used to trigger a script, it stays open for at least 15 minutes after the last script finishes.

Parallelism

At most 50 scripts can be run in parallel for the same user.

Synchronous execution (sync: true)

When sync is set to true in the script context file front matter, the UI waits up to 30 minutes for a response. During this time, no further interaction is possible.

  • If the script finalizes within those 30 minutes, the user sees the output of the script or any script errors that occurred.
  • If the script does not finalize within 30 minutes, it continues running in the background and errors are logged. However, the UI returns a message that no response was received within 30 minutes and that the script might still be running. No outputs, errors, or completeness events from the script are shown to the user.

Asynchronous execution (sync: false)

When sync is set to false in the script context file front matter, the script still runs synchronously in the background, but the UI returns immediately with a message that the script is queued for execution.

Enabling the feature flag

The following flag is introduced in the Assistant configuration. This can be added to the custom.appsettings.json file but will not appear in the default appsettings file.

As with other config settings, changes are picked up in real time without a DxM restart. However, the JSON files are not synced, so every appsettings file across all Assistant DxMs must be changed manually in a cluster.

"AssistantOptions": {
  "Insights": {
    "CanRunAutomationScripts": false
  }
}

The CanRunAutomationScripts option in the Insights block controls whether the native Insights agent can use Automation scripts.

  • Script discovery does not change.
  • When this flag is set to false, script tools are not provided to the native Insights agent.

This is a DxM-only change. The API remains unchanged.
